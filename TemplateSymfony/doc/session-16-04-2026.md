# Session du 16/04/2026 — Intégration Agent IA Coach Football

## Objectif de la session

Intégrer un agent IA spécialisé coaching football dans l'application Symfony existante (Andfield).
L'agent tourne en local sur Docker, utilise un LLM open-source, est développé en Python avec LangGraph,
et s'interface avec l'app Symfony via une API REST.

---

## Ce qui a été fait

### F1 — Infra Docker (LLM local)

**Fichiers créés / modifiés :**
- `compose.yaml` — ajout des services `ollama` et `agent` (remplacement de `localai`)
- `localai/qwen3-8b.yaml` — config modèle Qwen3 8B (référence, non utilisé en prod)
- `localai/qwen3-1.7b.yaml` — config modèle Qwen3 1.7B (référence, non utilisé en prod)

**Historique des tentatives LLM :**

| Tentative | Image | Modèle | Résultat |
|---|---|---|---|
| 1 | `localai/localai:latest-cpu` | Qwen3 8B Q4_K_M (5 Go) | **~3 min/réponse** — trop lent |
| 2 | `localai/localai:latest-cpu` | Qwen3 1.7B (HuggingFace) | **401 HuggingFace** — auth requise |
| 3 | `ollama/ollama:latest` | `qwen3:1.7b` | **~15s/réponse** ✅ |

**Solution retenue : Ollama**
- Image : `ollama/ollama:latest` (v0.20.7)
- Modèle : `qwen3:1.7b` Q4_K_M (~1.4 Go)
- API native Ollama sur `http://ollama:11434`
- Volume Docker `ollama_data` pour persister le modèle (pas de re-téléchargement)
- Modèle téléchargé automatiquement au premier démarrage via l'entrypoint

**Problème résolu — Thinking mode Qwen3 :**
Qwen3 utilise par défaut un mode "thinking" qui génère des tokens `<think>...</think>` avant
de répondre. Ce mode consommait tous les tokens alloués sans produire de réponse visible.

- L'API OpenAI-compatible Ollama (`/v1/chat/completions`) ne supporte **pas** `think: false`
- L'API native Ollama (`/api/chat`) supporte `think: false` → **1.6s**
- `langchain-openai.ChatOpenAI` + `extra_body={"think": False}` → **ignoré**
- `langchain-ollama.ChatOllama` + `reasoning=False` → **fonctionne** ✅ (~15s)

**Dépendance ajoutée :**
- `langchain-ollama>=0.2.1` (installé en v1.1.0) — paramètre `reasoning=False` supporté à partir de cette version

**Conventions :**
- Auth Symfony existante (session) — `#[IsGranted('ROLE_USER')]` sur toutes les routes

- Préfixe `/api/v1/`

**Authentification agent Python → Symfony :**
- `AgentTokenAuthenticator` : `Authorization: Bearer agent:<secret>` + header `X-Coach-Id`
- Placé dans le firewall `main` (et non un firewall `api` séparé) pour partager la session browser
- `AGENT_SECRET` partagé entre les containers `php` et `agent` via `.env`

### F4 — Agent Python LangGraph (core)

**Structure créée :**
```
agent/
  Dockerfile                  # Python 3.12-slim
  requirements.txt            # FastAPI, LangGraph, LangChain-Ollama, httpx
  app/
    __init__.py
    main.py                   # Serveur FastAPI + strip des tokens <think>
    config.py                 # Variables d'environnement
    memory.py                 # Couche persistance MySQL (F5)
    agent/
      __init__.py
      state.py                # AgentState (messages, coach_id, team_id, validation_mode, pending_action)
      prompts.py              # System prompt coaching football (commence par /no_think)
      tools.py                # 16 tools LangChain (CRUD + analyse)
      graph.py                # Workflow LangGraph — ChatOllama avec reasoning=False
```

**Workflow LangGraph :**
```
call_llm
  ├── (pas de tool call) → END
  ├── (tool call lecture) → execute_tools → call_llm
  └── (tool call écriture + mode singular) → request_validation
        ├── pending_action = None → apply_confirmed_action → END
        └── pending_action ≠ None → END (attente confirmation coach)
```

**Mode validation :**
- `singular` : l'agent propose l'action + attend `confirm_action: "confirm" | "cancel"`
- `auto` : exécute directement

**Tools disponibles :**
- `list/get/create/update/delete_team`
- `list/get/create/update/delete_player`
- `get_match_notes`, `create_match_note`
- `get_team_analysis` — agrégation joueurs + notes
- `suggest_composition` — propose une compo justifiée (F7)
- `coaching_report` — rapport complet de coaching (F7)
- `analyze_player` — analyse individuelle d'un joueur (F7)

**Endpoints FastAPI :**
- `GET  /health`
- `POST /chat`
- `GET  /conversations`

### F7 — Analyse & recommandations

**Fichiers modifiés :**

**Nouveaux tools :**

`suggest_composition(coach_id, team_id, formation?)` :
- Charge joueurs + notes post-match
- Groupe par position
- Extrait les mentions de chaque joueur dans les notes
- Retourne données structurées + instruction au LLM pour produire une compo justifiée

`coaching_report(coach_id, team_id)` :
- Charge équipe + joueurs + notes (8 dernières)
- Construit un map joueur → mentions dans les notes
- Retourne données + instruction structurée (5 sections : forme, méforme, collectif, axes, reco tactique)

`analyze_player(coach_id, team_id, player_id)` :
- Profil complet du joueur
- Toutes ses mentions dans les notes post-match
- Retourne données + instruction (évaluation titulaire/banc/surveiller)

**Enrichissement system prompt :**
- Commence par `/no_think` pour désactiver le mode thinking Qwen3
- Patterns à détecter (mention répétée = signal fort)
- Critères de sélection par ordre de priorité (position → forme → complémentarité → pied fort → physique)
- Instruction d'utilisation des tools d'analyse

## Bugs rencontrés et solutions

| Bug | Cause | Solution |
|---|---|---|
| CSS import 404 | Chemin `../styles/` invalide dans asset-mapper | Changé en `./styles/` |
| 401 sur les routes API depuis le browser | `fetch()` sans credentials | Ajout `credentials: 'include'` |
| 401 sur les routes API depuis l'agent Python | Firewall `api` stateless distinct du firewall `main` | `AgentTokenAuthenticator` déplacé dans le firewall `main` |
| `idle_timeout` non reconnu | Paramètre inexistant dans Symfony HttpClient | Remplacé par `max_duration: 600` |
| Timeout PHP 300s | `max_execution_time = 300` par défaut | `max_execution_time = 0` dans `10-app.ini` |
| LocalAI ne charge pas le modèle | Config YAML absente du volume Docker | `docker compose cp` ou bind-mount individuel |
| HuggingFace 401 sur modèle 1.7B | HF requiert authentification même pour repos publics | Abandon LocalAI → switch vers Ollama |
| Qwen3 thinking mode — `content` vide | Qwen3 consomme tous les tokens en `<think>` | `ChatOllama(reasoning=False)` — API native Ollama |
| `SERVER_PID` warning dans compose.yaml | `$SERVER_PID` interprété par Docker Compose comme variable d'env | Remplacé par `$$SERVER_PID` |
| Conflit de dépendances pip | `langchain-core==0.3.15` trop bas pour `langchain-ollama>=0.2.1` | Versions libres (`>=`) dans requirements.txt |

## Architecture finale déployée

```
Docker Compose
├── php (FrankenPHP + Symfony)  :80/:443  — App + API REST /api/v1/
├── database (MySQL 8)          :3306     — Données + conversations agent
├── ollama (Qwen3 1.7B CPU)     :11434    — LLM local (téléchargement auto au 1er start)
├── agent (Python FastAPI)      :8001     — LangGraph agent
└── mailer (Mailpit)            :8025     — Dev mail
```

```
Coach (navigateur)
  ↕ widget sidebar JS (credentials: include)
Symfony /api/v1/agent/chat
  ↕ httpx (timeout 600s, header X-Coach-Id)
Agent Python FastAPI :8001
  ↕ LangGraph graph
    ├── tools.py → /api/v1/* (CRUD Symfony, Bearer agent:<secret>)
    └── LLM → ChatOllama (reasoning=False) → Ollama :11434/api/chat (Qwen3 1.7B)
```
